# 3 万句分层场景题库扩容设计

## 1. 目标与范围

将离线听写填空应用的正式题库从 300 个英文句子扩展到 30,000 个不同英文句子。每个句子继续生成简单、中等、困难三个挖空版本，因此最终包含 90,000 道题。

本次同时完成以下改造：

- 场景从 4 个平铺类别扩展为 8 个大类、32 个子场景。
- 所有中文翻译均由自动翻译流程重新生成，不复用来源自带翻译。
- 构建过程支持断点续跑、失败重试、质量报告和原子发布。
- 应用按需查询题目，不一次性加载全部 90,000 道题。
- 保留现有 300 句的题目 ID、用户学习记录和未完成任务。

应用运行时继续完全离线。互联网访问、翻译模型和 LLM 只属于题库构建流程，不打进最终应用。

## 2. 已确认的产品约束

- 英文语料来源不受许可证类型限制，但必须保留来源信息。
- 内容适合家庭学习：排除色情、粗口、自残、仇恨、极端暴力和明显冒犯内容。
- 医疗、新闻和社会场景允许正常疾病、事故、法律等中性表达。
- 保留自然口语中的否定、争论、抱怨和情绪表达。
- 英文句子长度为 5 至 35 个英文单词。
- 中文翻译使用本地 OPUS-MT 初译，异常项由 LLM 修正，并由 LLM 分层抽检。
- 挖空答案支持单词和 2 至 4 词短语，空格数量必须等于答案单词数量。

## 3. 总体架构

题库生产拆成构建工作库与应用题库两层：

```text
互联网语料
  -> 原始语料工作库
  -> 清洗与内容安全过滤
  -> 场景分类、去重、配额筛选
  -> OPUS-MT 自动翻译
  -> 异常翻译交给 LLM 修正
  -> 生成简单/中等/困难三个挖空版本
  -> 质量门禁
  -> 原子发布 content.db
```

### 3.1 构建工作库

构建过程使用独立 SQLite 工作库，不再依赖单个大型 JSON 快照。工作库至少记录：

- 构建批次和阶段状态。
- 原始文本、来源名称、来源地址、作者和外部语料 ID。
- 清洗后文本、规范化哈希、SimHash 和拒绝原因。
- 大类、子场景、分类置信度和分类方式。
- 翻译文本、翻译引擎、模型版本、处理时间和质量状态。
- 三个题目版本及其难度依据。

每个阶段按批次提交，进程终止后从最后一个成功检查点恢复。已成功翻译或分类的句子不得重复调用模型。

### 3.2 应用题库

正式 `content.db` 只包含应用运行所需的数据：

- 大类与子场景元数据。
- 30,000 个最终句子。
- 90,000 个难度版本。
- 等价答案、来源信息和查询索引。

现有 300 句及其 `s0001-easy` 一类题目 ID 保持不变，只补充新的大类和子场景映射。新增句子使用规范化文本哈希生成稳定 ID，并在构建时检查哈希碰撞。

### 3.3 应用查询

内容仓库增加以下按需能力：

- 按大类、子场景和难度返回指定数量的候选题。
- 排除当前队列、已选择题号和调用方给出的题号。
- 按题号批量恢复未完成任务，不扫描全部题库。
- 无尽模式只维护当前题及预取窗口，不加载完整类别。

TTS 继续只生成当前题和后两题并缓存，不预生成全部音频。

## 4. 场景层级与配额

| 大类 | 子场景与句数 | 合计 |
|---|---|---:|
| 日常生活 | 家庭家务 1,500；社交沟通 1,800；购物服务 1,400；餐饮烹饪 1,300 | 6,000 |
| 出行旅行 | 交通通勤 1,200；问路导航 900；酒店住宿 1,100；旅行观光 1,300 | 4,500 |
| 职场商务 | 办公协作 1,300；会议演示 1,100；邮件电话 1,000；求职面试 1,100 | 4,500 |
| 学习考试 | 校园课堂 1,100；考试备考 900；学术研究 1,000；语言学习 1,000 | 4,000 |
| 健康医疗 | 医院就诊 800；药店用药 600；健身运动 800；身心健康 800 | 3,000 |
| 科技科学 | 数码设备 800；互联网软件 800；工程技术 700；科学科普 700 | 3,000 |
| 文化娱乐 | 影视戏剧 800；音乐艺术 700；阅读文学 700；体育休闲 800 | 3,000 |
| 新闻社会 | 时事新闻 600；财经商业 500；法律公共事务 400；环境社会 500 | 2,000 |

总计必须严格等于 30,000 句。构建器按子场景执行配额门禁，任何子场景不足都禁止发布。

## 5. 语料收集与来源控制

先收集至少 150,000 条候选，再筛选最终 30,000 条。候选来源包括：

- Tatoeba 详细句子数据。
- 对话与字幕语料。
- 新闻和公共事务文本。
- 文学与叙事文本。
- 技术、科学和教育文档。
- 任务型对话语料，例如酒店、餐饮、交通和客服对话。

Tatoeba 官方下载提供句子 ID、语言、文本、作者和修改时间，可用于稳定增量导入：<https://tatoeba.org/en/downloads>。

最终选择以场景配额和质量为准，不为单一来源设置固定产量，但需要限制单一来源和单一作者在每个子场景中的占比。构建报告必须输出来源、作者和外部语料 ID 的分布，以识别大量同结构人名句、字幕口头禅和单一作者偏置。

## 6. 清洗、过滤和去重

### 6.1 基础清洗

- 统一 Unicode、直引号、空白和 HTML 实体。
- 去除字幕时间码、说话人标签、舞台说明和孤立编号。
- 拒绝残句、多句拼接、乱码、不平衡引号和非英文主体文本。
- 句长必须为 5 至 35 个英文单词。

### 6.2 内容安全

拒绝色情、粗口、自残、仇恨、极端暴力和明显冒犯内容。正常医疗、事故、法律、争论和负面情绪不因主题本身被拒绝。

### 6.3 去重

- 规范化哈希负责精确重复。
- 64 位 SimHash 分桶产生近似重复候选。
- 仅在候选桶内执行词集 Jaccard 等精确相似度判断。

该方式替代当前所有句子两两比较的实现，避免 30,000 句规模下约 4.5 亿次比较。最终精确重复和达到阈值的近似重复均必须为零。

## 7. 场景分类

分类采用三级策略：

1. 来源显式标签或任务域映射。
2. 可复现的规则、关键词和文本特征分类。
3. 低置信度或多个场景分数接近时，由 LLM 在 32 个固定子场景中重判。

每个句子只设置一个主子场景。大类由子场景确定，避免同一句在配额统计中重复计数。分类规则和 LLM 提示词都必须带版本号。

## 8. 中文翻译

### 8.1 初译

使用 `Helsinki-NLP/opus-mt-en-zh` 对最终候选批量初译。该模型只在构建环境使用，不进入应用安装包。模型仓库：<https://huggingface.co/Helsinki-NLP/opus-mt-en-zh>。

### 8.2 自动校验

每条翻译检查：

- 非空且包含合理比例的中文字符。
- 英文数字、金额、日期和百分比未丢失。
- 人名、地名和大写专有名词没有无依据变化。
- 没有大段英文残留。
- 中英文长度比例和标点没有异常。

未通过任一检查的翻译进入 LLM 修正队列。

### 8.3 LLM 修正与抽检

- 所有自动校验异常项由 LLM 重新翻译。
- 每个子场景随机抽检至少 100 句，共至少 3,200 句。
- 抽检检查准确性、自然度、否定关系、数字、专有名词和场景语气。
- 抽检发现系统性问题时，按来源或错误类型扩大修正范围，而不是只修抽样句。

## 9. 挖空与难度

每个句子生成三个不同答案区间：

- 简单：优先选择常见实词或较短自然词组。
- 中等：选择中等词频实词、动词搭配或 2 至 3 词语块。
- 困难：选择低频词、固定搭配或 2 至 4 词高信息量语块。

难度分值综合词频、答案词数、固定搭配、句子长度和句法信息。质量约束：

- 答案必须是原句中的精确连续区间。
- `blank_count` 必须等于规范答案按空格拆分后的单词数。
- 同一句三个规范答案不得相同。
- 三个难度分值必须严格递增。
- 每个答案包含 1 至 4 个单词。
- 全库简单、中等、困难各 30,000 道。

## 10. 首页和练习页交互

- 首页使用 8 个紧凑大类按钮，分两行显示。
- 选中大类后显示该类的 4 个子场景及“全部该类”。
- 提供“全部场景”，可从完整题库抽题。
- 定量练习与无尽模式共用场景选择。
- 保存上一次选择的大类和子场景。
- 练习页顶部显示“大类 / 子场景”。
- 较窄窗口中场景区换行或独立滚动，不遮挡开始按钮。

大类和子场景来自数据库元数据，不再分别硬编码在 QML、Python 枚举和 SQLite `CHECK` 约束中。

## 11. 构建失败与发布

以下任一情况禁止覆盖正式 `content.db`：

- 任一子场景未达到确认配额。
- 总句数不是 30,000 或总版本数不是 90,000。
- 存在空翻译、答案区间错误、重复 ID 或哈希碰撞。
- 近似重复、内容安全或翻译质量门禁未通过。
- 数据库结构和索引完整性检查失败。

通过全部门禁后：

1. 生成质量报告和来源清单。
2. 在临时路径生成完整数据库。
3. 执行 SQLite 完整性检查和发布数据测试。
4. 以原子替换方式发布正式数据库。

失败构建保留工作库和报告，便于修复后继续，不影响当前可用的 300 句题库。

## 12. 质量报告

发布报告至少包含：

- 总句数、总题数和 32 个子场景配额。
- 三个难度的数量和分值分布。
- 句长、答案词数和来源分布。
- 作者集中度和人名集中度。
- 精确重复、近似重复和所有拒绝原因。
- 翻译引擎版本、LLM 修正数量和抽检结果。
- 数据库文件大小、查询耗时和完整性检查结果。

## 13. 测试与验收

### 13.1 构建管线

- 每个采集器、清洗器、分类器、翻译器、去重器和构建器都有单元测试。
- 断点续跑测试证明已经完成的翻译和分类不会重复执行。
- 使用缩小配额的集成数据完成端到端构建测试。
- 30,000 句正式数据运行全量发布门禁。

### 13.2 数据库和学习记录

- 验证 30,000 句、90,000 题和全部子场景精确配额。
- 验证现有 300 句题目 ID 仍可查询。
- 验证旧学习记录和未完成任务在新题库中可恢复。
- 验证按场景、难度、排除题号和题号批量查询。
- 验证定量练习和无尽模式不会一次性加载完整题库。

### 13.3 UI 和音频

- 验证 8 个大类与 32 个子场景均可选择。
- 验证大类、子场景和全部场景的筛选结果正确。
- 验证常见桌面窗口尺寸下按钮可见且页面可滚动。
- 验证自动播放、快捷键、TTS 预取和缓存策略不因题库扩容退化。

## 14. 非目标

- 不预生成或随安装包分发 30,000 句音频。
- 不在应用运行时联网翻译或下载语料。
- 不增加用户账号、云同步或在线题库服务。
- 不改变已经确认的三档难度和无尽模式升降级规则。
